## A. 답변 작성 양식 (필수)

각 질문 항목의 `- 영향:` 줄 **바로 아래**에 다음 한 줄을 추가합니다.

```
- 답변 (YYYY-MM-DD): 결정 내용 / 사유
```

- 날짜는 답변한 날짜 (예: `2026-05-08`).
- 결정 내용은 한두 줄로 압축. 길어지면 줄바꿈하여 동일 들여쓰기 유지.
- 답변이 다른 문서/스펙에 직접 들어갈 문구라면 **그대로 인용 가능한 형태**로 작성.

## B. 상태 토큰 변경 (권장, 누락해도 됨)

답변과 동시에 헤더의 상태 토큰을 변경:

- 답변 완료: `[질문중]` → `[답변완료]`
- 추가 논의 필요: `[질문중]` → `[재질문]`
- Phase 후순위: `[질문중]` → `[보류]`

귀찮으시면 그대로 두셔도 됩니다. 오케스트레이터가 일괄 정리합니다.

## C. 답변이 어려운 경우의 표기법

| 상황 | 표기 예시 |
|---|---|
| 아직 결정 못함 | `- 답변 (2026-05-08): 보류 — 베타 결과 후 결정` |
| 다른 결정에 의존 | `- 답변 (2026-05-08): C-01 결정 후 답변` |

## 0. 사용 방법

이 문서는 Sub agent들이 dash2zero 기획 문서를 검토하면서 발생시킨 질문을 모아 PM과 합의를 만들어가기 위한 단일 진실 원천(SSOT)이다.

### 흐름
1. 오케스트레이터가 sub agent에게 검토 대상 문서를 지시한다.
2. Agent가 문서를 읽고 질문을 만들어 자신의 섹션에 추가한다.
3. PM이 질문에 답변하고 상태를 변경한다.
4. 답변이 다른 문서/agent에 영향이 있으면 오케스트레이터가 "결정 사항 로그"에 정리한다.
5. 종결된 질문은 분기별로 아카이브 섹션으로 이동한다.

### 질문 항목 양식

```
#### Q-{agent}-{번호}. [상태] 질문 제목  (P0/P1/P2)
- 발견: 사업계획서 §X.Y 등
- 배경: 1-2줄
- 질문: 답변 가능한 단위 1개
- 영향: 답변이 안 되면 막히는 후속 작업

> **PM 답변 (YYYY-MM-DD):** ...
```

### 상태 표기
- `[질문중]` — 새 질문, 답변 대기
- `[답변완료]` — PM 답변 완료, 합의됨
- `[재질문]` — 답변에 대한 추가 질문이 있음
- `[보류]` — Phase 후순위로 미루기로 합의
- `[종결]` — 결정 완료, 후속 문서에 반영됨

### 우선순위
- **P0** — 출시 차단 또는 후속 문서 작성 차단
- **P1** — 출시 전 결정 필요하나 후속 문서 진행은 가능
- **P2** — 출시 후도 가능

- 영향: 운영 중 콘텐츠 수정 시 학습 데이터 무결성.
- 답변 (2026-05-07): 콘텐츠는 versioned immutable 원칙을 따른다. 단어 삭제 대신 retired_at을 사용하고, 기존 학습자의 word_id와 SRS 기록은 유지한다.

#### Q-BE-009. [답변완료] 음성 파일 호스팅 (P1)
- 발견: §9.3, §14.1
- 배경: 음성 파일 경로만 명시.

- 영향: 비용, 보안, 무단 사용.
- 답변 (2026-05-07): C-06과 동일. 음성 파일은 Supabase Storage public bucket 또는 signed URL 정책으로 제공하고 앱 캐시를 사용한다.

#### Q-BE-010. [답변완료] 무료/프리미엄 게이팅 위치 (P0)

#### Q-BE-011. [답변완료] 사용자 자정/타임존 (P0)

### 4.7 보안/개인정보 (Senior Security & Privacy) — 15개 질문

#### Q-SEC-001. [답변완료] GDPR/CCPA 적용 범위 및 법적 근거 (P0)
- 발견: §4.1, §17
- 배경: 영어권 1차 타깃이면 EU/EEA, UK, 캘리포니아 사용자가 사실상 포함됨.
- 질문: GDPR/UK GDPR/CCPA/CPRA를 처음부터 적용 대상으로 가정할지, 아니면 특정 지역 차단 또는 단계적 출시로 우회할지?
- 영향: 동의 배너, DSAR 응답, DPO/대리인 지정, 쿠키 정책 범위.
- 답변 (2026-05-07): C-10과 동일. MVP 출시 국가는 US/CA/UK/AU/NZ로 제한하고 EU 스토어프론트는 열지 않는다. 다만 UK GDPR/CCPA 수준의 열람·삭제·동의 철회 권리는 기본 제공한다.

#### Q-SEC-002. [답변완료] 수집 PII 항목과 합법 처리근거 매핑 (P0)
- 발견: §8.1, §10
- 배경: 온보딩에서 학습 목적·한국어 수준·관심 주제를 받지만 수집 항목 목록 부재.
- 질문: MVP에서 수집할 PII(이메일, 이름, 학습 동기, 한국어 수준, 디바이스 ID, IP, 광고 ID 등)와 각 항목의 법적 근거(consent/contract/legitimate interest)는?
- 영향: 처리방침 수집 항목 표, Privacy Label 정확성.
- 답변 (2026-05-07): 수집 PII는 이메일, 소셜 로그인 식별자, 구매 상태, 학습 데이터, 기기/분석 이벤트로 제한한다. 음성 녹음, 연락처, 위치, 광고 ID는 MVP에서 수집하지 않는다.

#### Q-SEC-003. [답변완료] 게스트 모드 vs 계정 가입 정책 (P0)
- 발견: §8.3
- 배경: 게스트 진입 허용 시 디바이스 ID는 GDPR/CCPA에서 PII 취급.
- 질문: 게스트 모드 식별자(디바이스 ID, 광고 ID, UUID)와 보관 기간, 계정 전환 시 데이터 병합 규칙은?
- 영향: 게스트 삭제권 행사 흐름, 익명성 수준.
- 답변 (2026-05-07): C-04와 동일. 게스트는 개인정보 최소화를 위해 로컬 저장만 사용한다. 서버 동기화, 구매, 데이터 권리 행사는 계정 전환 후 제공한다.

#### Q-SEC-004. [답변완료] 미성년자 처리 (COPPA / GDPR 16세 미만) (P0)
- 발견: §4.1, §3.1
- 배경: K-콘텐츠 동기 학습자는 13–17세 비중 높음. COPPA/GDPR은 부모 동의 필수.
- 질문: 가입 시 연령 확인 게이트, 13세 미만/16세 미만 차단 또는 부모 동의 흐름 중 무엇인가? 앱스토어 연령 등급 목표는?
- 영향: 미준수 시 앱스토어 반려, FTC/EU DPA 제재.
- 답변 (2026-05-07): MVP는 어린이 대상 서비스가 아니며 13세 미만은 가입과 결제를 차단한다. 온보딩 또는 계정 생성 시 “I am 13 or older” 확인을 둔다. EU 출시 전에는 16세 기준 정책을 재검토한다.

#### Q-SEC-005. [답변완료] 계정/데이터 삭제 및 익스포트 흐름 (P0)
- 발견: §12.3, §8.1

- 질문: 앱 내 계정 삭제 버튼 위치, 처리 시간(즉시/유예), 삭제 범위, 데이터 익스포트(JSON/CSV) 제공 여부는?
- 영향: Apple 심사 통과, DSAR SLA.
- 답변 (2026-05-07): C-11과 동일. 앱 내 계정 삭제와 이메일 요청을 모두 지원한다. 삭제 SLA는 30일, 데이터 내보내기 SLA도 30일이다.

#### Q-SEC-006. [답변완료] 데이터 보관 기간 정책 (P0)
- 발견: §8.1
- 배경: GDPR/한국 PIPA는 항목별 보관 기간 명시 필수. 결제 영수증은 세법상 5년.
- 질문: 계정/학습 상태/퀴즈 이력/결제 영수증/푸시 토큰/분석 이벤트/로그 각각의 보관 기간과 파기 방식은?
- 영향: 처리방침 보관·파기 섹션, 라이프사이클 자동화.
- 답변 (2026-05-07): 학습 데이터는 계정 유지 기간 동안 보관한다. 계정 삭제 시 30일 내 삭제한다. 결제/세무 기록은 법적 필요 기간 동안 최소 정보만 보존하며 상세 기간은 법무 문서에서 확정한다.

#### Q-SEC-007. [답변완료] 분석 SDK / 결제 SDK 데이터 송신과 동의 (P0)
- 발견: §12.1, §14.1
- 배경: Firebase Analytics/RevenueCat/Adjust 등은 IP·디바이스 ID·광고 ID를 미국으로 전송. EU에서는 동의 전 비활성화 필수.
- 질문: 사용 SDK 목록과 EU 사용자 대상 동의 전 차단(consent gating) 정책, ATT 프롬프트 필요 여부는?
- 영향: 국외이전 동의, ATT 미준수 시 Apple 반려, Privacy Manifest 정확성.
- 답변 (2026-05-07): 분석 SDK는 Firebase, 결제 SDK는 RevenueCat으로 제한한다. 광고 SDK는 사용하지 않는다. 비필수 분석은 동의 설정을 제공하고, 제3자 전송 항목은 처리방침에 명시한다.

#### Q-SEC-008. [답변완료] 백엔드 위치와 국외 이전 (P1)
- 발견: §14.1
- 배경: 영어권 사용자 데이터가 한국/미국 서버에 저장되면 GDPR Chapter V 적용.
- 질문: 백엔드/DB/CDN 호스팅 리전과 국외 이전 방식(SCC, 적정성, 동의)은?

#### Q-SEC-009. [답변완료] 결제 영수증/구독 상태 처리 (P1)

#### Q-SEC-010. [답변완료] 음성 파일 저작권 및 사용권 (P1)

#### Q-SEC-011. [답변완료] 앱스토어 Privacy Label / Data Safety / Privacy Manifest (P0)

#### Q-SEC-012. [답변완료] 푸시 알림 토큰 및 동의 (P1)

- 발견: §4.1, §9.3
- 배경: A0/A1을 "완전 초보 또는 초급"으로만 기술하고 단어 `level` 필드에 A0 사용.
- 질문: A0/A1의 조작적 정의(어휘 수, 문장 길이)와 TOPIK 1급/세종학당 레벨 매핑표를 후속 커리큘럼에 둘 것인가?
- 영향: 매핑이 없으면 단어 선정과 예문 i+1 판단 불가능.
- 답변 (2026-05-07): MVP 레벨은 내부적으로 A0, A1, A1+만 사용한다. TOPIK 직접 매핑은 하지 않고 “TOPIK 전 단계/초급 준비” 참고 라벨만 둔다.

#### Q-LD-002. [답변완료] SRS 알고리즘 선택 (P0)
- 발견: §7.2, §8.1
- 배경: "적절한 시점에 다시 만난다"고만 기술되고 알고리즘 미정의.
- 질문: Leitner / SM-2 / FSRS / 단순 고정 간격 중 무엇이며 초기 간격, 정답/오답 시 간격 조정 계수는?
- 영향: 복습 큐 구현, ERD, D7 리텐션 KPI.
- 답변 (2026-05-07): C-08과 동일. 자체 Leitner 5단계 SRS를 채택한다.

#### Q-LD-003. [답변완료] "Mastered" 정량 기준 (P0)
- 발견: §13.2
- 배경: "일정 기준 이상 정답률과 복습 이력"이라는 정성 표현만.
- 질문: 정답률 X% 이상 / Y회 연속 정답 / 마지막 복습 후 N일 등 구체적 임계값은?
- 영향: 진도/리포트/유료 가치.
- 답변 (2026-05-07): Mastered는 SRS stage 5 이상, 누적 정답률 80% 이상, 최소 4회 정답 기록을 모두 만족한 단어다. Weak는 최근 5회 중 오답 2회 이상 또는 14일 이상 overdue인 단어다.

#### Q-LD-004. [답변완료] 300단어의 빈도 기반성 (P0)
- 발견: §9.1, §9.2
- 배경: 주제별 100/100/100 배분만 있고 한국어 빈도 코퍼스 근거 없음.
- 질문: 빈도 상위 단어와 주제 적합성 단어의 비율은? 빈도 기준 출처는?
- 영향: 단어 선정의 임의성이 콘텐츠 신뢰도를 좌우.
- 답변 (2026-05-07): 300단어는 빈도만으로 선정하지 않는다. 기준 가중치는 실사용성 40%, 초보 난이도 25%, 여행/K-콘텐츠 적합성 20%, 예문 확장성 15%다.

#### Q-LD-005. [답변완료] 한글 자모 선행 학습 (P0)
- 발견: §3.1, §7.2
- 배경: 한글을 처음 보는 사용자가 단어 카드로 바로 진입.
- 질문: 자모 14+10 또는 음절 블록 학습 모듈을 온보딩에 포함하는가, 단어 학습에 자모 인식을 녹이는가?
- 영향: 첫 학습 완료율, D1 리텐션.
- 답변 (2026-05-07): 한글 자모 선행 코스는 필수로 분리하지 않는다. 온보딩 후 첫 5개 카드에서 자모/음절 구조를 아주 짧게 노출하고, 이후 단어 카드 안에서 점진적으로 반복한다.

#### Q-LD-006. [답변완료] 로마자 표기 체계 선택 (P0)
- 발견: §9.3
- 배경: "sagwa"는 RR에 가까우나 표기 체계가 명시되지 않음.
- 질문: RR / MR / IPA / 자체 표기 중 무엇을 표준으로 하며, 받침·격음·경음 규칙은?
- 영향: 300단어 일관성, 검수 0/1 기준.
- 답변 (2026-05-07): C-18과 동일. 로마자 표기는 Revised Romanization을 기본으로 한다. 발음상 초보자 혼동이 큰 경우 pronunciation_note 필드로 보완한다.

#### Q-LD-007. [답변완료] 조사·표현의 단위 처리 (P0)
- 발견: §9.3
- 배경: "주세요"가 단어인지 표현인지, 조사가 단어 카드인지 패턴인지 불분명.
- 질문: 단어 단위 정의(자립명사/표현/조사)와 300단어 안에 표현·조사 비중은?
- 영향: "300단어"의 의미와 카드 UX 구조.
- 답변 (2026-05-07): 조사와 표현은 단어와 분리된 “pattern” 단위로 관리한다. MVP는 주세요, 이에요/예요, 은/는, 을/를, 에, 에서 같은 최소 패턴만 예문 보조로 다룬다.

#### Q-LD-008. [답변완료] 발음 변화 표기 정책 (P1)
- 발견: §3.1
- 배경: 연음, 받침 발음(좋아요 → [조아요])을 어떻게 보여줄지 미정.
- 질문: 카드에 한글 표기·발음 표기·로마자를 모두 노출하는가, 음성으로만 처리하는가?
- 영향: 카드 정보 밀도와 초보자 인지 부담.
- 답변 (2026-05-07): 발음 변화는 MVP에서 과도하게 설명하지 않는다. 단어/예문에 실제 발음 음성을 제공하고, 필요한 경우 short pronunciation note만 표시한다.

#### Q-LD-009. [답변완료] 다의어·다품사 분해 정책 (P1)
- 발견: §9.3
- 배경: "주다/주세요", "오다/와요"처럼 한 어휘에 여러 형태/의미가 있을 때 카드 분해 기준 없음.
- 질문: 한 단어 = 한 카드 원칙인가, 활용형은 분리 카드인가?
- 영향: 300단어 카운팅과 distractor 설계.
- 답변 (2026-05-07): 다의어/다품사는 sense 단위로 분리한다. 초보 단계에서는 가장 실용적인 의미 1개를 기본으로 노출하고 다른 의미는 Phase 3 이후 확장한다.

#### Q-LD-010. [답변완료] 음성 품질 표준 (P1)
- 발견: §8.1, §14.1
- 배경: TTS인지 사람인지, 화자 일관성·속도·억양 기준 없음.
- 질문: 단어/예문 음성의 화자 수, 발화 속도, 한 사람 한 권/다화자 정책은?
- 영향: Hear 단계 신뢰도, 음성 비용, 재제작 부담.
- 답변 (2026-05-07): 음성 품질은 44.1kHz 또는 48kHz 원본에서 mp3/aac로 배포하고, 단어 음성은 2초 이내, 예문은 5초 이내를 기준으로 한다. 잡음 없는 TTS 파일만 승인한다.

#### Q-LD-011. [답변완료] 오답 후보(distractor) 설계 규칙 (P1)
- 발견: §9.3
- 배경: 같은 카테고리 단어를 distractor로 사용할 뿐 의미적/형태적/발음적 거리 기준 없음.
- 질문: distractor 3개 중 의미/형태/무관 비율은? 자동 생성 시 정량 검증 기준은?
- 영향: 퀴즈 난이도 일관성, 정답률 KPI 해석.
- 답변 (2026-05-07): 오답 후보는 같은 품사 또는 같은 카테고리에서 고르되 정답과 지나치게 유사해 초보자가 학습 불능이 되지 않게 한다. 동일 세션 내 중복 오답은 피한다.

#### Q-LD-012. [답변완료] 예문 i+1 적합성 (P1)
- 발견: §7.2, §9.3
- 배경: "사과 주세요"처럼 1단어 예문도 있고 더 긴 예문도 가능. 통제 기준 없음.
- 질문: 한 예문의 최대 음절/단어 수, 신규 단어 외에 등장 가능한 어휘 범위는?
- 영향: 초보자 좌절 시 Context 단계 작동 불가.
- 답변 (2026-05-07): 예문은 i+1 원칙을 따른다. 신규 단어 외 문장 구성은 이미 학습했거나 A0-A1 수준으로 제한하고, 한 예문은 3-6어절 이내로 한다.

#### Q-LD-013. [답변완료] AI 검수 정량 기준 (P1)
- 발견: §9.4
- 배경: 자연성·정확성·문화 적합성이 정성적 항목으로만 나열.
- 질문: 각 항목을 0/1 또는 5점 척도로 정량화하고 통과 임계값(예: 자연성 4/5 이상)을 둘 것인가?
- 영향: 1인 운영에서 재현 가능한 콘텐츠 품질.
- 답변 (2026-05-07): AI 생성 초안은 사람 검수 전에는 배포하지 않는다. 검수 체크는 의미 정확성, 자연성, 초보 난이도, 문화 적합성, 로마자/발음 일치 5개 항목을 모두 통과해야 한다.

#### Q-LD-014. [답변완료] 마스터드 단어 장기 재노출 (P1)
- 발견: §13.2, §16
- 배경: Mastered 후 큐에서 사라지면 1-2개월 내 망각 위험.
- 질문: Mastered 단어를 30/60/90일 간격 재노출 정책이 있는가? 신규 큐와의 비율은?
- 영향: 장기 보존, 300단어 완주 후 동기.
- 답변 (2026-05-07): Mastered 단어도 장기 재노출한다. stage 5 이후 30일, 60일, 120일 간격으로 가볍게 재출제하고 오답 시 stage 3으로 하향한다.

#### Q-LD-015. [답변완료] 300단어 완주 후 학습 동선 (P0)
- 발견: §7.1
- 배경: §16 Phase 3에서 콘텐츠 팩 확장이 언급되나 MVP 출시 시점의 완주자 동선 미정.
- 질문: MVP에서 300단어 소진 시 (a) 추가 팩 (b) 무한 복습 (c) 대기 화면 중 무엇인가?
- 영향: D30+ 리텐션과 연간 구독 정당화.
- 답변 (2026-05-07): C-19와 동일. 300단어 완주 후에는 Review Garden, Weak Words, 월 50개 신규 팩, 카테고리별 챌린지를 제공한다.

#### Q-LD-016. [답변완료] 게임화와 학습 효과 충돌 (P2)
- 발견: §13.1, §8.1
- 배경: 스트릭이 끊기지 않으려는 동기가 능동 인출보다 "탭만 하기"로 흐를 수 있음.
- 질문: 스트릭/배지가 단순 출석이 아닌 "복습 정답"에 묶이는 정책을 도입할 것인가?
- 영향: 학습 효과 KPI(정답률, Mastered) 신뢰도.
- 답변 (2026-05-07): 스트릭과 배지는 단순 앱 오픈이 아니라 lesson_completed 또는 review_completed에 묶는다. 최소 1개 이상의 능동 인출 정답이 있어야 streak가 유지된다.

### 4.10 데이터/분석 (Senior Data & Analytics) — 15개 질문

#### Q-DA-001. [답변완료] 분석 도구 스택 미확정 (P0)
- 발견: §12.1, §14.1
- 배경: "분석 이벤트는 적게 시작" 외 도구가 명시되지 않음.

- 영향: 도구별 형식·카디널리티·식별자·가격이 다름.

#### Q-DA-002. [답변완료] D1/D7 리텐션의 측정 트리거 정의 (P0)
- 발견: §13.1
- 배경: "다음 날 다시 온 비율"이 앱 오픈/학습 시작/학습 완료 어느 것인지 불명확.
- 질문: D1/D7의 retain 정의는 `app_open`/`lesson_started`/`lesson_completed` 중? Calendar day(UTC/로컬)인가, rolling 24h인가?
- 영향: 동일 코호트 D1이 2-3배 차이 가능.
- 답변 (2026-05-07): D1/D7 retain 정의는 lesson_started 기준이다. 날짜 기준은 사용자 로컬 타임존의 calendar day이며 일일 리셋은 04:00이다.

#### Q-DA-003. [답변완료] "하루 3분 루프"의 측정 이벤트 (P0)
- 발견: §7.1, §13.1
- 배경: 세션 시작/종료/이탈 이벤트 정의 없음.
- 질문: `session_started` 트리거가 앱 진입인가 학습 화면 진입인가? 세션 종료는 명시적 `lesson_completed`인가, 백그라운드 N분 후 타임아웃인가? `lesson_abandoned`는 잡는가?
- 영향: 평균 세션 길이는 종료 정의에 매우 민감.
- 답변 (2026-05-07): session_started는 학습 세션 화면 진입 시, lesson_completed는 신규 3단어 루프 완료 시 발사한다. 앱이 5분 이상 백그라운드면 lesson_abandoned로 기록한다.

#### Q-DA-004. [답변완료] Mastered/Weak Words 임계값 (P0)
- 발견: §13.2
- 배경: "일정 기준 이상 정답률과 복습 이력"이라 정성적.
- 질문: Mastered = 정답률 ≥ X%이며 SRS 단계 ≥ N? Weak = 최근 K회 중 오답 ≥ M회 또는 미복습 ≥ D일? 클라이언트/서버 어디서 계산?
- 영향: 학습 품질 KPI 보고 일치성.
- 답변 (2026-05-07): Q-LD-003과 동일. Mastered는 stage 5+, 정답률 80%+, 최소 4회 정답이다. Weak는 최근 5회 중 오답 2회 이상 또는 14일 이상 overdue다.

#### Q-DA-005. [답변완료] 게스트→가입 사용자 머지 (identify) (P0)
- 발견: §8.3, §10.3
- 배경: 게스트 진입 후 학습 가능, 이후 가입/결제 발생 가능.
- 질문: 게스트 디바이스 ID 이벤트를 가입 시 server-side identify로 머지하는가? 머지 실패 시 동일 사용자 2명 계산 위험을 어떻게 막는가?
- 영향: 모든 핵심 KPI(전환율, 리텐션, LTV)가 부정확.
- 답변 (2026-05-07): C-04와 동일. 게스트 device_install_id는 가입 시 user_id와 alias/identify로 연결한다. 병합 성공 후 게스트 ID는 동일 사용자의 이전 익명 ID로만 보존하고 중복 사용자로 집계하지 않는다.

#### Q-DA-006. [답변완료] 사용자 식별자 정책 (P0)
- 발견: §8.1, §10
- 배경: 결제는 IAP 기반이지만 계정 시스템 명시 없음.

- 영향: 분석 키, 결제 복구, GDPR 삭제 처리 모두 결정.
- 답변 (2026-05-07): C-03과 동일. 서버 1차 식별자는 Supabase auth user_id다. Apple private relay 이메일은 일반 이메일과 동일하게 취급하되 표시명/마케팅 목적 사용은 하지 않는다.

#### Q-DA-007. [답변완료] 결제 퍼널 이벤트 분기 (P0)

#### 옵션 A — Cross-cutting P0-2 (C2-01 ~ C2-20)부터 답변 (권장)

#### 옵션 C — 보류 + PRD 우선 작성

#### 옵션 D — 답변 종결 / 후속 문서로 전환
