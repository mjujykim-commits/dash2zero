## 1. 임무

R-12("SRS sibling copy drift 위험")의 영구 해소. readiness §6에서 권고한 옵션 비교를 ADR-0006로 봉인 + Phase 1 패키지 스켈레톤 작성.

## 2. 결정 요약

> **Option A 채택 — `packages/srs-core` 신규 워크스페이스 패키지 + tsup ESM/CJS dual 빌드.**

### 핵심 논리

1. **비대칭 해소**: Payment는 W14에 `_shared/billing.ts` 단일 SoT로 정리됐으나 SRS는 sibling 그대로. architecture 일관성 훼손.
2. **evaluator 신뢰 토대**: SRS golden 50 + RLS adversarial 9가 W15-W16에 활성화되는 시점, "검증 대상 = 운영 대상" 등식이 깨지면 Layer 4 Evaluation의 alpha→beta 전환 불가.
3. **Boring stack 부합**: tsup은 esbuild 위 얇은 wrapper, 신규 SaaS 0개. 운영 컴포넌트 ≤ 5 원칙 유지.

### 거부된 대안

- Option B (코드 생성): Deno-incompatible API 자동 변환 불가, fragile.
- Option C (parity-check): signature/상수만 비교 — 본문 의미 변화 미감지. 임시방편.

### Phase 분리

| Phase | 시점 | 작업 주체 | 게이트 |

| 1. 패키지 분리 | W15 (본 ADR과 동시) | architect (스켈레톤) | tsup build 통과, ESM/CJS dist 산출 |
| 2. import 통합 | W16 | backend | mobile spec + golden runner 양측 green |
| 3. sibling 제거 + R-12 closed | M4 W17 | backend | RISK REGISTER 갱신 |

## 4. 영향 범위 (다른 agent에 대한 의존 정리)

- **backend (W15-W16)**: Phase 2 import 통합 작업 큐. 현재 W15에 RLS evaluator + Payment fixtures 진행 중이므로 SRS 본 구현 이전은 W16 우선순위. 본 ADR은 그 작업의 사양서 역할.
- **security**: 영향 없음 (도메인 분리). 단 SRS evaluator green이 보안 회귀 검증과 무관함을 확인.
- **content / learning**: 영향 없음. SRS 도메인 규칙(C-08 / CC2-09 / CC2-10 / CC3-05 / CC-17)은 변경 없음 — 패키징만 바뀐다.
- **mobile (Phase 2 작업 시)**: `apps/mobile/package.json`에 `"@dash2zero/srs-core": "workspace:*"` 추가, metro workspace resolver 검증 (이미 contracts에서 검증됐으면 NOOP).
- **CI/CD**: workflow에 `pnpm -F @dash2zero/srs-core build` step + Supabase deploy 전 dist sync step 추가 필요 (Phase 2).
